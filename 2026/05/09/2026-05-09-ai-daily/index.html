Claude Code v2.1.136：MCP 生态修复与 IDE 集成的关键一跃

背景：为什么这个话题重要

Claude Code 在 5 月 8 日发布的 v2.1.136 版本，虽然版本号只比前日 +3，但修复的问题触及了 AI 辅助编程的基础设施层——MCP（Model Context Protocol）服务器管理和 IDE 插件集成。对于将 Claude Code 嵌入 VS Code 或 JetBrains 工作流的开发者来说，这次更新解决了三个长期痛点：MCP 服务器在 /clear 后神秘消失、多服务器并发刷新时的 OAuth Token 竞争、以及扩展思考模式下的 API 400 错误。这些问题看似是”小 bug”，实则直接影响生产环境中 AI 工具链的稳定性。

原理：技术实现的核心逻辑

MCP 服务器持久化修复 解决了一个隐蔽的状态同步问题。当用户在 VS Code 扩展或 JetBrains 插件中执行 /clear 时，Claude Code 会重置会话上下文，但此前这个操作会连带清除 .mcp.json、插件配置和 claude.ai 连接器中注册的 MCP 服务器信息。v2.1.136 将 MCP 配置与会话状态解耦，确保 /clear 只清理对话历史，不碰工具链配置。

OAuth Token 并发刷新修复 则是一个经典的竞态条件问题。当用户配置了多个远程 MCP 服务器（如 GitHub、Slack、Notion 等），每个服务器都有自己的 OAuth 刷新令牌。此前多个服务器同时刷新时，后完成的写入会覆盖先完成的，导致先刷新的服务器令牌丢失。新版本的修复逻辑是：在刷新期间对令牌存储加文件级锁，确保并发刷新串行化，同时保留每个服务器的独立刷新状态。

扩展思考 400 错误修复 涉及 Anthropic API 的 thinking block 协议。当 Claude 在扩展思考模式下生成工具调用后，如果 thinking block 被标记为 redacted（因安全策略隐藏），后续 API 请求会携带一个格式错误的 thinking 字段，导致 400 错误。新版本在序列化请求前增加了 redacted block 的过滤逻辑。

场景二：多 MCP 服务器的 OAuth 配置

如果你有多个需要 OAuth 的 MCP 服务器（如 Google Drive、Slack、Notion），建议：

  1. 逐个添加并认证，避免并发认证
  2. 认证完成后，检查 ~/.claude/mcp-auth/ 目录下的令牌文件是否各自独立
  3. 如果遇到”需要每日重新认证”的问题，升级到 v2.1.136 后应彻底解决

场景三：扩展思考模式的安全使用

适用场景

适合用：

  • 在 VS Code/JetBrains 中深度集成 Claude Code 的团队
  • 使用多个 MCP 服务器构建复杂 AI 工作流的开发者
  • 依赖扩展思考模式进行架构设计、代码审查等深度任务

不适合用：

  • 仅使用命令行版本且未配置 MCP 服务器的用户（这些改动对你无感知）
  • 使用 Claude Code 进行简单问答、不涉及工具调用的场景
